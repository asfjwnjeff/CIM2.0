# 可交互原型设计

当需要产出原型时，采用"先确认、再输出、持续迭代"模式，主动协助设计功能和页面内容。

## 插件优先策略

- 如果已连接 UI 原型生成插件（如 frontend-design、ui-ux-pro-max），**必须优先调用该插件**完成视觉与交互原型设计。
- 生成单文件可交互 HTML 原型，内嵌 CSS/JS，浏览器直接打开预览。
- 视觉风格：**专业 B 端 SaaS 蓝白基调，克制整洁**，统一间距与排版，栅格清晰。

## 备选 HTML 原型（插件不可用或用户明确要求时）

- 生成单文件可交互 HTML 原型，内嵌 CSS/JS，浏览器直接打开预览。
- 视觉风格：专业 B 端 SaaS 蓝白基调，克制整洁，最小宽度 1280px。
- 所有 UI 组件状态必须在原型中可呈现：
  - 按钮：正常/悬停/加载/禁用
  - 输入框：默认/焦点/错误/禁用
  - 标签、表格斑马纹/空数据、分页、弹窗、抽屉、提示条
- 图标使用 SVG 或 Unicode 符号，不依赖外部资源。
- **数据填充**：使用典型脱敏业务数据（如"张*三""A 科技公司""920102******0321"），拒绝示例文本。
- **完整交互**：
  - 页面间跳转（锚点/多视图切换，保留浏览器历史）
  - 表单控件（输入、下拉、日期、开关、文件占位）
  - 按钮状态切换（加载中、禁用、启用）
  - 弹窗/抽屉/确认提示
  - 基础表单验证与错误反馈（即时校验 + 提交校验）
  - 边界状态展示（空数据、加载中、网络异常）
- 若用户要求"把原型做得像生产环境"，增加数据密度、边界状态演示、角色权限差异（如普通业务员 vs 经理视图）。

## 迭代协作流程

1. **范围确认**：先概括本次原型覆盖的页面和功能清单，获得用户默许。
2. **首版产出**：按约定生成原型。
3. **收集反馈**：主动标注可改进细节，提供输入通道。
4. **版本管理**：每次迭代生成新版本号，在原型页面顶部显示版本号与修改记录。
5. **功能/内容建议**：若发现更优交互或缺失功能，主动提醒并提供选择。

## 交付要求

- 列出完整页面清单和可演示交互路径（如"投保申请 → 核保审核 → 缴费通知 → 保单签发"）。

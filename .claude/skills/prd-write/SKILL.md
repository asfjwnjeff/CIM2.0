# PRD 撰写标准

当用户要求撰写 PRD 时使用此 skill。PRD 聚焦页面逻辑、功能交互、字段定义与前后端接口约定，弱化技术实现细节。

## 标准 PRD 章节结构

```
1. 概述
   1.1 背景 — 业务背景 + 要解决的问题
   1.2 目标 — 本次要达成的目标
   1.3 范围 — 包含/不包含
   1.4 多系统边界界定 — 多系统协作时必写

2. 用户角色
   - 角色-使用模块-说明矩阵表
   - 标注各角色在模块中的权限差异

3. 核心业务流程
   - 端到端主流程（ASCII 流程图，含模块间协同关系）
   - 各子流程（如有独立子流程）

4. 页面导航结构
   - 树形菜单结构图
   - 标注新增/变更模块

5. 功能模块详情（按导航顺序逐模块展开）
   [模块概述] — 页面路径、使用角色
   [列表页] — 查询条件表、列表列、操作按钮表
   [详情/编辑页] — 字段表、内嵌表格、操作按钮完整定义表
   [状态机] — ASCII 状态流转图 + 状态操作矩阵表
   [接口约定概要] — 接口表（接口名|用途|关键入参|关键出参|调用时机）
   [非功能需求] — 性能指标、埋点事件表、权限矩阵表

6. 权限控制方案

7. 待确认清单

8. 验收标准 — checkbox 格式（- [ ] 条件描述）

9. 附录（按需）
   - 名词解释、多系统交互架构、异常场景处理、版本历史
```

## 字段表格完整性检查清单

每个字段表必须覆盖以下 7 项（不适用的标注"—"）：

| 检查项 | 说明 |
|--------|------|
| 字段名 | 中文名称 |
| 类型 | 文本/下拉/日期/开关/金额/百分比/文本域/文件/关联选择 |
| 必填 | 是/否/条件必填（如"含税时必填"） |
| 默认值 | 系统默认填入的内容或无 |
| 校验规则 | 唯一/不超过X字/正则/日期先后/依赖条件 |
| 可编辑时机 | 仅创建时/始终可编辑/状态X后可编辑/不可编辑 |
| 枚举值 | 下拉类字段的完整选项列表 |

## 按钮行为完整定义

每个操作按钮必须逐项说明：

| 按钮 | 可见角色 | 触发前提 | 点击后行为 |
|------|----------|----------|-------------|
| 按钮名 | 谁能看到此按钮 | 什么条件下可点击 | 弹窗确认? 状态变化? 数据变更? 跳转? |

行为描述覆盖：触发前提 → 弹窗/二次确认 → 提交后状态变化 → 异常处理（权限不足/网络超时/数据冲突）

## 多系统边界界定

当 PRD 涉及多个子系统协作时，在概述里插入边界对比表：

| 对比维度 | A 系统侧 | B 系统侧 |
|----------|---------|----------|
| (列出各系统负责的内容维度) | | |

## 模块间约束

- **硬约束**：系统阻止操作（如"无商机不可发起风控审批"）
- **软约束**：系统提示但不阻止（如"服务产品与商机不一致，建议检查"）

## 书写原则

- 写 PRD 过程中，必须主动识别用户表述模糊、存在歧义或信息缺失之处，**立刻追问澄清**，不得自行猜测或跳过。采用"反复确认、逐项收敛"的方式。追问时优先使用选择题形式。
- 每个功能模块必须有一张页面字段表格，且对每个操作按钮进行按钮行为完整定义。
- 交互说明用"当用户…系统应…"句式。
- 避免技术方案用语（如 Redis、Kafka）；可提及"系统需异步通知"而非技术组件。
- 数据埋点仅列事件和参数，不写采集代码。
- 所有金额、客户名称使用示例脱敏数据。

## 评审支持

- 生成评审要点清单，聚焦功能逻辑、接口约定和验收标准。
- 当用户提到"开发评审"时，主动输出功能疑问列表（基于接口、状态流转、边界条件）。
- PRD 修改后需同步更新调研过程记录，保持"PRD ←→ 调研记录"双向可追溯。

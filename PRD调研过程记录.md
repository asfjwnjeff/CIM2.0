# CIM系统V3.0 PRD调研过程记录

**版本**: v2.0
**日期**: 2026-06-12
**说明**: 本文档记录了CIM系统V3.0 PRD的完整调研过程。v2.0 新增附录：CIM2.0 系统字段规格（完整字段定义、类型、必填、可选值）。

---

## 调研方法说明

本次调研遵循 CLAUDE.md 中定义的引导式设计框架：
1. **场景与角色梳理** — 用户角色 + 核心使用场景
2. **流程推演** — 端到端业务流转 + 决策点确认
3. **信息架构与字段补全** — 页面清单 + 字段列表
4. **交互与状态细化** — 按钮行为、状态流转、异常分支

每轮只问2-4个问题，优先用选择题降低用户回答成本。每轮确认后总结共识，再进入下一轮。

## 第一阶段：场景与角色梳理

### 问题

> 这三个页面（字典管理、账单拆分字段管理、账单主体规则管理）主要由谁来使用？

### 用户回答

> "字典管理是IT和后台运维使用，前台人员主要是客服/客服主管/经理，他们需要去操作配置和查看账单拆分字段管理，和账单主体规则管理。财务其他页面（最多是查看），可能也会协助客服配置主体判断规则。"

> "首次配置后改动应该只是偶尔的，业务变化/客户要求变化才改，但是我没办法一次性配置所有客户，我需要他们自己慢慢用这个系统封去配置好，首次推广周期应该不短。"

> "数据权限必须管控，分客户的。功能权限有大概的管控。管控我们有统一权限平台，你帮我设计怎么管控就行，系统功能是现成的。"

### 用户原始需求叙述

> "我这个原型还有很多问题，我需要通过反复询问你协助我修改，先从PRD开始。你根据当前的这个原型的内容先帮我搞个框架然后开始询问我。基础页面是差不多的，但是增改查的页面不太符合要求。"

> "菜单栏里的主体管理不对，其实是客户主体管理，比如结算主体，签约主体，服务主体（客户）。"

> "规则管理页面是要做账单拆分用的，逻辑呢是是这样的，涉及三个系统，COS（订单系统），CIM（客户信息管理系统），CPQ（正式报价录入，自动计费，生成账单）。然后COS下单的时候，客服需要录入一些字段（当前这些字段是输入框，所以五花八门），例如部门，货主区域，货物属性，但是我需要在CIM做一个配置，这样配置过的客户在下单时，对应字段就变成下拉框，可选值范围就是在CIM中配置好的。这样当该订单完成后，进入CPQ计费（报价之前已维护），CPQ取CIM中的字段判断规则，比如某个客户的规则是当货主区域是上海，且部门是设备部，那就是计入账单1。其他还有规则（按客户设立）就能把每一笔订单拆分到对应账单中。并且所有数据都是标准化的，不存在数据混乱的问题。"

> "我设计分成三个页面。先是字典管理，里面有CIM系统中所有的字段，包括字典管理的基础功能，然后还要有一个字段分类比如工商信息字段，产业链字段，以及账单拆分字段，其他的可选值在字典管理中配置，以及字段类型：下拉单选，多选，输入框，日期等。花园字段名称、字段标识、是否必填，字段说明。账单拆分字段的可选值在账单拆区分段管理中配置，按客户维度配置，编辑页面内容就是客户，字段名称，可选值，可选值做成点加号减号增减一行一行的，输入框。账单主体规则管理，就配置判断规则，支持且和或，以及组内嵌套，可视化简单操作，按客户维度分别配置，账单主体的名称是手动输入，必填，下面规则条件的地方是做成下拉框单选字段和值的。条件有等于，不等于，包含，不包含，为空，不为空等，还有一个备注说明，可选输入，添加规则说明。要有操作日志。"

### 第11.1轮：场景与角色

**用户核心表述：**

> A,B,C,D都参与。AB在我们公司是并为一个角色。

> 主管/经理发起报价，给到客服，客服去发给客户看，然后客户对报价有问题也只是告知客服，客服作为中转给到经理/主管。修改价格和折扣等都是主管/经理等领导层决定。最后如果成单，就让客服走风控流程审批，再签合同。之后就把数据接入CPQ，继续正式报价的录入实现自动计费等。

> PDF和H5主要是看客户接受程度，有的需要发PDF，有的就要链接。

> CPQ目前是没有商机的。新的这个售前报价，我要设计成非绑定的。L1L2L3的标准名必须遵守。

**确认结论**：经理/主管（定价权）→ 客服（中转发送）→ 客户（H5查看确认）；不绑定商机；费用名称必须遵循L1/L2/L3标准产品目录。

### 第11.2轮：流程推演

**用户核心表述：**

> 版本改成报价轮次。加上是否含税，税率，我们公司主体选择（必填），部门，报价发起人名称，还有申明。计费单位是可以选择的，有标准的，比如票，件，个，项，料号...

> 导出PDF也需要有模板，按版式导出的。客户看到报价单是有版式的，这个版式是我们配好的。

> 客户那里也会有折扣，比如多轮价格商谈下来，最后定了一个价格，然后客户说给我总价打个95折，那这个时候需要能显示折扣。明细的费用也会有这种情况，比如运输的费用打个95折。

> 还需要把备注传到CPQ，备注是很重要的，备注讲清楚了费用如何计费。

> 合同签署之后衔接CPQ。CPQ是最全的报价数据...但是售前报价不需要这么复杂，只需要能让客户看清楚费用是怎么收的，不需要考虑让系统实现自动计费。

> 是不是需要有一份类似菜单一样的标准报价模板（但不能太随意暴露自己的价格），然后再给定制化的报价？

> 我有很多业务，一个客户刚进来我并不知道要做什么业务，肯定是全都报给他。但后面真要做可能就挑几个业务去做...可以分业务搞模板，这样如果知道对方需要的业务我可以直接选择对应的业务模板发过去。

> 可以给客服看中间每次的价格，因为作为中转，也需要给客户解释价格怎么收取。但是底层模板可以设定权限是否给客服看。

**确认结论**：报价轮次+快照机制；双层级折扣（行级+整单）；备注核心字段同步CPQ；按业务类型分模板（仓储/运输/关务/其他）；模板级客服可见权限控制；头信息新增含税/税率/公司主体/部门/发起人/声明；PDF按版式模板渲染。

### 第11.3轮：信息架构与字段设计

**用户核心表述：**

> Q6不需要（数量字段）。Q8先按A来（一套通用版式）。Q9 A（每发出一轮=一个快照，轮内多次修改不增加轮次，只记修改日志）。有默认声明，但是每家客户/业务可以手动修改。

**确认结论**：售前只需展示单价；PDF一套通用版式；每发送=快照轮次，轮内修改只记字段级日志；声明默认文本可手动修改。

### 第11.4轮：交互与状态细化

**用户核心表述：**

> Q11需要的，确认之后才能开始走风控。Q12继续改就行。H5链接跟随报价有效期，并且当版本更新后，老轮次的报价链接就失效了。

**确认结论**：客户H5确认 → 客服系统内二次确认 → 状态变更 → 方可走风控；重新激活延续旧报价不强制新轮次；H5链接跟随有效期，新轮次触发旧链接失效。

## 第十二阶段：模块架构与边界重定义（2026-05-12）

**用户提供了完整的新流程图，一次性提出多项结构性变更，从根本上重新定义了 CIM 系统边界。**

### 用户核心表述

> 首先我只需要CIM的。售前报价还是叫售前报价。只是流程变了：线索→客户→商机→报价→方案→风控→定制化报价→生效→合同签署。标准报价必须在客户有商机的情况下才能发出。加一个"方案管理"模块，方案和报价是协同看的一起发给客户。定制化报价是个性化议价。CPQ，CIM只是售前报价，售前报价完了才去CPQ成交计费，CIM不需要合同了。

> 有一个重要的点，所有的商机，风控，合同，报价都需要选择服务产品：货代、关务、仓储、运输、进出口、维修、合同物流、一体化供应链、其他。一个客户商机和风控，合同的服务产品需要一致。

> 这个约束是这样的，一个客户可以多个商机，只是它从一个商机往下走的时候，必须和这个商机的服务产品一致。可以先做成软约束，不要强制约束。

### 用户对方案管理的补充

> 方案管理就是合同物流服务方案书，里面要覆盖站点、业务量、KPI要求、是否贸易代理、审批流程。有标准方案和定制方案。

### 用户对客户主体的补充

> 客户拆成三类主体：服务主体、签约主体、结算主体，每个都是多对多关系，需要手动绑定。结算主体有银行账号和开户行。

### 确认结论

| 议题 | 结论 |
|------|------|
| 售前报价绑定商机 | **标准报价必须有商机才能发出**（推翻第11.1轮"不绑定商机"结论） |
| 方案管理 | **新增模块**，与售前报价并行；售前报价解决"收多少钱"，方案解决"怎么服务" |
| 定制化报价 | 个性化议价阶段，对报价进行定制化修改 |
| 报价生效 | 方案和报价确定后一并生效，作为 CIM 侧终点 |
| CIM/CPQ 边界 | CIM 结束于报价生效；合同签署、计费规则、自动计费、账单、开票、回款全部在 CPQ 侧 |
| 合同模块 | **从 CIM 中完全移除**（取代第九阶段的设计） |
| 服务产品 | 9 项标准枚举，作为 CIM 核心字段贯穿商机→报价→方案→风控 |
| 服务产品约束 | 从商机往下走时，下游模块的服务产品必须与该商机一致；**软约束**（提示不强制）；一个客户可有多商机，各商机链服务产品独立 |
| 客户主体重构 | 客户是大概念 → 下挂三类独立主体（服务/签约/结算），均为多对多关系，手动绑定（取代第三阶段"服务主体=客户本身"的结论） |
| 主流程 | 线索→客户→商机(必须)+服务产品→售前报价+方案管理→风控审批(绑定商机)→定制化报价修改→确定生效→[CIM结束→CPQ:合同签署→计费规则→自动计费→账单→开票→回款] |

### 本阶段推翻/修正的历史决策

| 历史决策（阶段） | 修正后 |
|-----------------|--------|
| 售前报价不绑定商机（第11.1轮） | 标准报价必须绑定商机 |
| 服务主体=客户本身，不需单独管理（第三阶段） | 服务主体独立管理，与客户 M:N 绑定 |
| 合同签署在CIM侧（第九阶段） | 合同模块整体移除，移至CPQ侧 |
| 签约主体+结算主体（第三阶段） | 扩展为三类主体：服务/签约/结算 |
| 商机→售前报价→风控→CPQ（第八阶段） | 新增方案管理并行、定制化报价阶段、明确报价生效为CIM终点 |

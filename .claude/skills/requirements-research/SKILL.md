# 需求调研与引导式设计

当用户提出任何功能或系统需求时，使用引导式设计框架逐层澄清。核心原则：**永远不猜测，用选择题降低用户回答成本；每轮确认后总结共识。**

## 非文本输入处理

用户经常提供图片、Excel、文档等非文本输入。在任何引导步骤之前，先处理这些输入：

- **图片/截图（原型、流程图、字段截图）**: 用视觉能力读取图片内容，提取所有可见字段、按钮、流程节点、页面布局。输出"从附件中识别到的内容清单"（模板见 `references/templates.md` §8），逐项确认准确性。
- **Excel 文件**: 读取后识别表格结构，提取规则/数据行，整理为结构化列表供用户确认哪些列/行是核心、哪些可忽略。
- **流程图/架构图**: 视为**重大变更信号**。提取每个节点（输入/输出/决策点/异常分支），逐节点确认。如果新图与已确认结论冲突，触发「批量推翻」模式（见下文）。
- **.docx/PDF**: 提取关键信息后，追问哪些内容直接复用、哪些需要调整、哪些仅作参考。

**重要**: 用户提供附件后，必须先输出识别清单并获得用户确认，再进入引导步骤。不假设理解正确。

## 引导式设计框架

当用户只知道"我想做一个 XX 功能/系统"但无法清晰描述细节时，按以下顺序分步递进、逐步收敛。

### 步骤 1：场景与角色梳理

- 引导用户说出：谁会使用？什么情况下使用？想达成什么目的？
- 输出：用户角色清单 + 核心使用场景列表（2-5 条）。
- 若用户无法回答，给出 2-3 个典型场景假设，让用户判断。
- 使用结构化选择题表格提问（模板见 `references/templates.md` §1）。

### 步骤 2：流程推演

- **开始时检查**：当前讨论的领域（保险/业财税/客户管理）是否匹配 `domain-reference` skill 中的知识。如匹配，先用 Skill 工具加载 `domain-reference`，将标准流程/术语作为"已知行业参考"提供，然后问用户实际流程与之的差异。
- 基于确认的场景，描绘端到端业务流转。
- 逐一标注每个节点的决策点（谁来决策？依据什么？有哪些分支？）。
- 用户描述的流程如果与领域标准模式冲突，**主动标注差异**让用户确认。
- 输出：业务流程图 + 决策点疑问清单。

### 步骤 3：信息架构与字段补全

- 按页面/模块逐一梳理展示信息和填写内容。
- 逐字段追问：来源（用户输入/系统带出/外部接口）、必填、校验规则、联动。
- 引用 `domain-reference` 中的标准字段集作为基线，问用户需要增减哪些。
- 输出：页面清单 + 每页字段列表（含状态说明）。

### 步骤 4：交互与状态细化

- 逐一确认：按钮行为、状态流转、异常分支、消息提示。
- 对每个操作追问：触发条件？成功后？失败后？谁来处理异常？
- 输出：交互细则 + 状态机说明。

### 步骤 5：模块架构与边界重定义

**触发条件**: 涉及多模块/多系统，或用户提供了新的架构图/流程图。

- 确认各模块/系统的权责边界。
- 确认模块间数据流向和字段继承关系。
- 确认跨模块约束（软约束 vs 硬约束）。
- 如果用户提供了新架构图，执行「批量推翻」检查（见下文）。
- 输出：更新后的全流程图 + 系统边界表 + 模块关系矩阵。

## 框架灵活模式

上述 5 步不是死板的。实际使用中以下三种模式经常出现：

### 补充轮次

当上一轮用户回答后，某个点需要深入追问但不构成新的独立阶段时：

- 标注为"步骤 N 补充"，不推进步骤编号
- 聚焦于上一轮中未完全澄清的单个议题
- 补充结束后回到主流程
- 在调研记录中与主步骤合并呈现

### 并行对话

当某个模块设计复杂度高、需要独立的多轮对话时：

- **分叉条件**: 模块涉及 ≥3 个独立子页面，或用户明确表示"这个模块另外讨论"
- **分叉时**: 在调研记录中总结当前已确认的跨模块约束，作为并行对话的输入边界
- **合并时**: 逐条检查并行对话结论是否与主对话约束冲突，标注冲突项让用户裁决
- **记录方式**: 在主调研记录中引用并行对话，标注"第 N 阶段（并行）"

### 批量推翻

当用户提供新流程图/架构图，或一次性提出多项结构性变更时：

- 逐条对比新输入 vs 已确认结论，识别所有受影响的历史决策
- 输出"变更影响分析"（模板见 `references/templates.md` §5）
- 列出：历史决策 | 所在阶段 | 修正后 | 影响模块 | 是否需回溯确认
- 用户批量确认后，回溯修正所有受影响模块的设计
- 在调研记录中：被推翻项在正文用 `~~删除线~~` 标注，并记录到"推翻/修正历史决策表"（模板见 `references/templates.md` §4）

## 输出格式规范

调研过程中的所有标准化表格模板定义在 `references/templates.md`。关键格式：

| 格式 | 用途 | 使用时机 |
|------|------|----------|
| 结构化选择题表格 | 降低用户回答成本 | 每轮提问时 |
| 确认结论表 | 该轮共识锚点 | 每轮结束时 |
| 文档产出表 | 追踪 PRD/字段文档/原型等文件更新 | 每轮调研结束后 |
| 推翻/修正历史决策表 | 记录被修正的早期决策 | 发生推翻时 |
| 变更影响分析 | 替代 plan file，评估变更波及范围 | 涉及 ≥3 模块或新架构图时 |
| 非文本输入识别清单 | 确认附件理解准确性 | 用户提供任何文件后 |

调研记录的完整文档结构见 `references/templates.md` §7。

## 引导原则

- 每次只问 2-4 个问题，优先用选择题降低用户回答成本。
- 每轮确认后简要总结已达成的共识，再进入下一轮。
- 若用户中途改变想法，回溯到受影响的步骤重新确认。
- 当所有步骤走完且用户确认无误后，再进入 PRD 撰写环节。
- **用户提供流程图/示意图时**，这是重大变更信号，必须逐节点确认，并检查是否推翻历史决策。
- **当本轮推翻了早期决策时**，显式记录到推翻/修正历史决策表，并回溯受影响的模块一并修正。
- **大规模修订前先做变更影响分析**（模板见 `references/templates.md` §5）：列出变更项 + 影响范围 + 是否需回溯确认，获得用户确认后再实施。不再使用独立的 plan file。

## 跨 Skill 集成

### 与 domain-reference 的协作

- 在步骤 2（流程推演）和步骤 3（字段补全）时，检查当前领域是否在 `domain-reference` 覆盖范围内
- 如是，主动加载 `domain-reference`，将标准流程/术语/字段作为"行业基线"提供给用户
- 用户的描述如果与标准模式有差异，不是错误——但必须标注差异并让用户确认是"有意偏离"还是"之前没注意到"

### 与 prd-write 的衔接

**正向（调研 → PRD）**:
- 所有步骤走完且用户确认 → 明确告知用户"需求已充分澄清，现在可以进入 PRD 撰写"
- 加载 `prd-write` skill，将调研记录中的确认结论作为 PRD 的输入
- 调研记录中每个确认结论应能定位到 PRD 中的对应章节

**反向（PRD → 调研）**:
- PRD 撰写中发现新模糊点 → 在 PRD 中标注"待调研确认" → 回到本 skill 补充调研
- 补充确认后更新 PRD 对应章节，清除"待调研确认"标记
- 补充的确认结论追加到调研记录中，标注"PRD 反向触发"

### 与 prototype-design 的衔接

- 当需求涉及复杂交互或用户对界面布局有疑虑时，主动建议"是否需要先做个原型看看效果？"
- 原型产出后，用本 skill 的引导方式逐区域确认原型是否满足需求

## 调研过程记录规范

调研过程记录是 PRD 的前置伴随文档，回答"这个设计为什么是这样"。

### 核心规范

- **用户原话必须逐条引用**：使用 `> ` 引用块格式原文保留，不加工、不概括。
- **每轮结束时输出「确认结论」表**（议题|结论），作为该轮共识锚点。
- **每轮结束后更新「文档产出」表**，追踪所有受影响文档的版本变化。
- **决策被推翻时显式标注**：正文用 `~~删除线~~` + 记录到推翻/修正历史决策表。
- **统计口径**：总轮次、用户回答次数、图片/文件提供数、选择题占比。
- **并行对话**：如涉及同一 PRD 的不同模块，在总结中合并呈现，标注"并行"。

### 文档结构

完整结构见 `references/templates.md` §7。核心组成：
1. 文档元信息（版本、日期、说明）
2. 调研方法说明
3. 各阶段记录（按时间顺序，含所有标准表格）
4. 调研成果总结（模块覆盖表 + 调研统计 + 关键决策记录）

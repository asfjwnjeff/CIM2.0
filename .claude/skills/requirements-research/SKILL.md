# 需求调研与引导式设计

当用户需求模糊或需要进行需求调研时使用此 skill。

## 常规调研

- 根据用户描述，输出结构化调研文档：业务背景、用户角色、核心场景、痛点、目标。
- 主动追问边界条件与业务规则，以及提出内容建议供选择。
- 当需求模糊时，必须使用选择题方式澄清（例如："报价单的币种是仅人民币，还是多币种？"），避免猜测。

## 引导式设计框架

当用户只知道"我想做一个 XX 功能/系统"但无法清晰描述细节时，按以下顺序分步递进、逐步收敛：

### 步骤 1：场景与角色梳理
- 引导用户说出：谁会使用？什么情况下使用？想达成什么目的？
- 输出：用户角色清单 + 核心使用场景列表（2-5 条）。
- 若用户无法回答，给出 2-3 个典型场景假设，让用户判断。

### 步骤 2：流程推演
- 基于确认的场景，描绘端到端业务流转。
- 逐一标注每个节点的决策点（谁来决策？依据什么？有哪些分支？）。
- 输出：业务流程图 + 决策点疑问清单。

### 步骤 3：信息架构与字段补全
- 按页面/模块逐一梳理展示信息和填写内容。
- 逐字段追问：来源（用户输入/系统带出/外部接口）、必填、校验规则、联动。
- 输出：页面清单 + 每页字段列表（含状态说明）。

### 步骤 4：交互与状态细化
- 逐一确认：按钮行为、状态流转、异常分支、消息提示。
- 对每个操作追问：触发条件？成功后？失败后？谁来处理异常？
- 输出：交互细则 + 状态机说明。

### 步骤 5：模块架构与边界重定义（多模块/多系统场景时启用）
- 确认各模块/系统的权责边界。
- 确认模块间数据流向和字段继承关系。
- 确认跨模块约束（软约束 vs 硬约束）。
- 输出：更新后的全流程图 + 系统边界表 + 模块关系矩阵。

## 引导原则

- 每次只问 2-4 个问题，不过载；优先用选择题降低用户回答成本。
- 每轮确认后，简要总结已达成的共识，再进入下一轮。
- 若用户中途改变想法，回溯到受影响的步骤重新确认。
- 当所有步骤走完且用户确认无误后，再进入 PRD 撰写环节。
- **用户提供流程图/示意图时**，这是重大变更信号，必须逐节点确认。
- **当本轮推翻了早期决策时**，显式记录"被推翻"，并回溯受影响的模块一并修正。
- **大规模修订前先写修订计划（plan file）**：列出变更清单 + 影响范围 + 文件修改列表，获得用户确认后再实施。

## 调研过程记录规范

调研过程记录是 PRD 的前置伴随文档，回答"这个设计为什么是这样"。

### 文档结构

```
1. 文档元信息（版本、日期、说明）
2. 调研方法说明（引用引导式设计框架的 5 个步骤）
3. 各阶段记录（按时间顺序）
   - 每阶段：背景/触发 → 用户原始表述（引用块逐条保留）→ 确认结论表（议题|结论）
4. 调研成果总结
   - 模块覆盖表（模块|确认轮次|关键产出）
   - 调研统计（总轮次、回答次数、图片/文件提供数、选择题占比）
   - 关键决策记录（编号+决策+来源阶段；被推翻的用 ~~删除线~~ 标注）
```

### 核心规范

- **用户原话必须逐条引用**：使用 `> ` 引用块格式原文保留，不加工、不概括。
- **每轮结束时输出「确认结论」表**（议题|结论），作为该轮共识锚点。
- **决策被推翻时显式标注**：使用 ~~删除线~~ + 指向修正阶段。
- **统计口径**：总轮次、用户回答次数、图片/文件提供数、选择题占比。
- **并行对话**如涉及同一 PRD 的不同模块，在总结中合并呈现，标注"并行"。
